## 三、关键技术决策

| 决策点 | 选择 | 原因 |
|--------|------|------|
| **PDF 展示** | 预转 PNG 图片 | 加载快、移动端兼容好、无需 PDF.js |
| **主部署平台** | GitHub Pages | 免费、稳定、与仓库一体，免备案 |
| **构建流程** | 根目录 `npm ci` → `pdf-to-images` → `cd app` → `build` | 支持 monorepo，PDF 转图在构建前完成 |
| **basePath** | `/shenxianzhongzi` | 适配 GitHub Pages 子路径 |
| **部署前检查** | Lint（提示）+ Build（必须通过） | 降低远程构建失败率 |
| **部署反馈** | watch-build 轮询 GitHub API | 无需 gh CLI，跨平台可用 |

## 六、经验与教训

1. **多平台部署**：初期同时配置 Cloudflare、Vercel、Netlify，导致 watch 脚本误报；最终收敛到 GitHub Pages 为主，简化维护。
2. **PDF 方案**：从 iframe → PDF.js → 预转图片，每次迭代都解决一部分问题，最终预转图片在加载速度和兼容性上最均衡。
3. **子模块**：损坏的子模块配置会导致 Cloudflare 克隆失败，需彻底移除引用。
4. **basePath**：GitHub Pages 子路径影响 manifest、资源、PDF 链接，需统一用相对路径或 `basePath` 处理。
5. **部署规范**：部署前 build 检查 + 推送后 watch 反馈，显著减少「推送成功但线上失败」的情况。
